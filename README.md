# Yuri Rewrite

Yuri Rewrite 是一款 Windows 本地桌面工具，用于导入 TXT 小说，调用用户自行配置的 AI 模型完成内容分析，并将小说改写为双女主 / 百合版本。

小说正文、改写结果、模型配置、日志和导出文件都保存在本机。项目不会提供公共 AI 服务，使用前需要准备自己的 API Key。

## 下载与运行

1. 打开 GitHub 仓库的 [Releases](https://github.com/3minto1/Yuri-Rewrite/releases) 页面。
2. 下载名称类似 `YuriRewrite-v...-windows-x64.zip` 的 portable 压缩包。
3. 将压缩包完整解压到一个可写目录。
4. 双击其中的 `Yuri Rewrite.exe` 启动程序。

程序目前没有代码签名。Windows SmartScreen 如果显示“未知发布者”，请确认文件来自本仓库的 Releases 页面，再选择“更多信息”并继续运行。

## 开始前准备

你需要准备：

- Windows x64 系统。
- UTF-8、GBK 或常见中文编码的 TXT 小说文件。
- 一个可用的 AI 服务账号及 API Key。
- 对应服务的 Base URL 和模型名称。
- 稳定的网络连接，以及足够的模型额度。

支持两类模型接口：

- OpenAI-compatible Chat Completions：适用于 OpenAI、DeepSeek、Moonshot/Kimi、SiliconFlow 以及其他兼容服务。
- Gemini：使用 Google Gemini 原生接口。

不同服务的模型名称、地址、计费和并发限制可能变化，请以服务商当前文档为准。

## 快速上手

### 1. 导入小说

点击左侧的“导入 TXT”，选择小说文件。程序会自动识别常见的中文章节标题；无法识别时，会按正文长度自动分段。

导入完成后，小说会出现在左侧列表中。章节小说默认每 30 章组成一个批次，非章节文本按固定长度组成批次。

### 2. 添加模型

在模型配置区域新建模型，填写：

| 字段 | 说明 |
| --- | --- |
| 提供商 | 选择 OpenAI-compatible 或 Gemini |
| Base URL | AI 服务的接口地址 |
| 模型名称 | 服务商提供的模型 ID，不是自定义显示名称 |
| API Key | 仅保存在本机，不会写入 AI 日志 |
| 思考模式 | 不确定时使用“自动”；模型不支持时可关闭 |

保存后先点击“诊断模型”。诊断会检查连接、API Key、基础回复、JSON 输出和思考模式兼容性，不会改写小说内容。

### 3. 填写小说设定

选择小说并打开“小说设定”，至少填写原主角姓名。你还可以配置：

- 改写后的主角姓名。
- 需要一起女性化的其他姓名。
- 主角体型和胸部设定。
- 严格改写或创意改写模式。
- 只对当前小说生效的高级要求。

小说设定与当前小说绑定，不会影响其他小说。

### 4. 选择批次

一本小说可能包含多个批次。先在批次列表中选择要处理的范围，再选择分析、改写或一键运行。

### 5. 分析原文

“分析”只提取原作中的剧情、人物、性别、关系、称谓、地点、伏笔和名称等一致性信息，不会提前注入百合改写要求。

### 6. 执行改写

分析完成后可以改写当前批次。程序会结合小说设定、全局核心设定和一致性资料生成改写稿。

需要自动完成当前批次时，可使用“一键分析改写当前批次”；需要依次处理整本小说时，可使用“一键分析改写”。全文任务支持暂停、继续和终止。

### 7. 对比和导出

改写完成后进入“对比”页面，可逐章查看原文和改写稿：

- 全局搜索会同时搜索全部章节的原文与改写稿。
- 差异高亮会标出删除、替换和新增内容。
- 可以向上或向下跳转搜索结果。
- 导出只包含已经改写完成且内容非空的章节，不会用原文补齐失败章节。

## 任务类型怎么选

| 操作 | 适用场景 | 执行内容 |
| --- | --- | --- |
| 分析 | 第一次处理某个批次 | 只分析原文并生成一致性资料 |
| 改写 | 当前批次已经分析完成 | 只改写当前批次中可处理的章节 |
| 一键分析改写当前批次 | 想处理选中的一个批次 | 先分析，再改写当前批次 |
| 一键分析改写 | 想自动处理整本小说 | 按批次分析、改写并导出 |

模型请求可能耗时较长。提高并发通常能缩短等待时间，但也会增加瞬时请求数、限流概率和失败风险。服务不稳定时建议降低并发。

## 应用设置

应用设置对所有小说生效：

- **导出目录**：保存批次文件和完整改写 TXT 的位置。
- **核心设定**：每次改写都会携带的全局风格要求，不适合填写某一本小说的具体剧情。
- **双专家复检**：让另一个模型检查改写结果，发现问题时最多要求改写模型重新生成两次。
- **复检模型**：可单独指定；未指定时使用当前改写模型。
- **并发数**：分析和改写共享的请求并发设置。

开启复检会显著增加请求次数、等待时间和模型费用。

## 本地数据与隐私

- 小说和应用数据保存在系统应用数据目录的 SQLite 数据库及配套文件中。
- API Key 优先写入 Windows 系统凭据库。
- 系统凭据库不可用时，程序会明确提示正在使用数据库兜底存储。
- AI 日志可以包含提示内容、模型回复和服务商原始响应，但不会记录 API Key 或 Authorization 请求头。
- 调用在线模型时，相关小说片段会发送给你所选择的 AI 服务商，请先确认其隐私政策。

删除小说前，程序会列出将一同删除的数据并要求确认。确认后会清理章节、批次、设定、一致性资料、改写稿、任务记录、相关日志和内部批次文件。删除模型时也会尝试删除对应的本地凭据。

重要小说请自行保留原始 TXT 和已导出的改写文件。

## 常见问题

### API Key 保存后仍然诊断失败

先查看模型列表中的凭据状态和诊断结果。常见原因包括 Base URL 填错、模型名称不存在、Key 权限不足、额度耗尽或服务商拒绝当前地区的请求。诊断结果会尽量保留服务商返回的错误信息。

### JSON 或思考模式诊断出现警告

部分兼容接口并不支持标准 JSON 参数或思考参数。可以将思考模式设为“自动”或“关闭”；程序只会在服务商明确表示参数不兼容时进行兼容重试。

### 出现章节 marker 或输出截断错误

程序使用稳定的章节开始和结束标记把批量回复拆回每一章。模型输出不完整时会自动重试，并尝试缩小分片范围。若最终仍失败，请查看 AI 日志，降低并发，或换用上下文和最大输出能力更强的模型；程序不会把无法可靠解析的内容写入章节。

### 服务商提示内容过滤

这通常是模型服务商的安全策略拦截，不是章节解析失败。请根据日志中的服务商响应调整模型、提示要求或服务商配置。

### 任务长时间没有完成

单次请求最长可能等待较长时间。先观察任务进度和 AI 日志；频繁出现限流或网络错误时降低并发。暂停或终止全量任务后，可以从第一个未完成批次继续。

### 对比页差异高亮为什么会降级

为了避免超长文本或高差异文本阻塞界面，程序会在计算过慢或高亮区间过多时改用行级差异；极端情况下只显示普通文本。搜索、原文、改写稿和导出内容不会因此改变。

### 项目目录里的 `src-tauri/target` 很大

Rust 调试编译会保存大量依赖和增量缓存。它们不是源代码，也不会提交到 Git。开发者可以在项目根目录运行安全清理脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\clean-debug-cache.ps1 -DryRun
powershell -ExecutionPolicy Bypass -File .\clean-debug-cache.ps1
```

脚本只清理 Rust 的 debug/dev 构建缓存，并检查 release 产物没有被改变。清理后下一次调试编译会重新构建依赖，因此会更慢。

## 开发者指南

### 环境要求

- Node.js 和 npm
- Rust stable 工具链
- Tauri v2 所需的 Windows 构建环境与 WebView2

### 常用命令

```powershell
# 安装前端依赖
npm install

# 启动 Tauri 开发模式
npm run tauri:dev

# 运行前端测试
npm test

# 构建前端
npm run build

# 运行 Rust 测试
cargo test --manifest-path .\src-tauri\Cargo.toml

# 严格检查 Rust 代码
cargo clippy --manifest-path .\src-tauri\Cargo.toml --all-targets --all-features -- -D warnings

# 构建 Windows 应用
npm run tauri:build

# 生成 portable 压缩包
npm run package:portable
```

portable 文件会生成到：

```text
portable/YuriRewrite-v{version}-windows-x64.zip
```

`dist/`、`node_modules/`、`portable/` 和 `src-tauri/target/` 都是生成目录，不应提交到 Git。

## 项目结构

```text
src/                         React 前端
  components/                工作区、设置、对比页和公共组件
  hooks/                     小说、模型和任务状态 hooks
  store/                     Zustand 运行时状态
  types/                     集中管理的 TypeScript 类型
  tauriApi.ts                强类型 Tauri 命令封装

src-tauri/                   Rust / Tauri 后端
  src/commands/              小说、模型、任务、设置、导出等命令
  src/ai/                    OpenAI-compatible、Gemini、提示词和解析
  src/db/                    SQLite schema 与迁移
  src/text/                  编码识别与章节切分
  src/lib.rs                 应用初始化、共享编排与命令注册

scripts/package-portable.ps1 portable 打包脚本
clean-debug-cache.ps1        安全清理 Rust debug 缓存
```
