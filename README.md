# Yuri Rewrite

Yuri Rewrite 是一款 Windows 本地桌面工具，用来导入 TXT 小说，调用你自己配置的 AI 模型分析原文，并把符合条件的章节改写为双女主 / 百合版本。它支持批量分析改写、分片并发、暂停恢复、一致性资产、单章重写、复检、差异对比和 TXT 导出。

小说正文、改写稿、模型配置、任务日志、Token 统计、导出文件和 API Key 都保存在本机。程序本身不提供公共 AI 服务，使用前需要准备自己的模型账号和 API Key。

## 下载与运行

1. 打开 GitHub 仓库的 [Releases](https://github.com/3minto1/Yuri-Rewrite/releases) 页面。
2. 下载 `YuriRewrite-v...-windows-x64.zip` portable 压缩包。
3. 解压到一个可写目录。
4. 双击 `Yuri Rewrite.exe` 启动。

程序目前没有代码签名。Windows SmartScreen 如果提示“未知发布者”，请确认文件来自本仓库 Releases 页面，再选择继续运行。

标准 portable 目录支持应用内更新。点击“检查更新”后，程序会下载并校验最新 ZIP，短暂关闭软件，在同级目录生成新版本文件夹，然后自动重新打开。GitHub 下载长时间没有进展时会依次尝试内置镜像；全部失败后会提示前往 Releases 页面手动下载。自动安装失败会尝试恢复并重新启动旧版本。

只有从 `YuriRewrite-v...-windows-x64` 标准目录运行且 GitHub 提供 SHA-256 摘要时才会自动替换程序。开发环境、目录不可写或缺少摘要时只下载 ZIP，由用户手动解压。更新只替换程序文件，不会删除应用数据目录中的小说、设置、日志、Token 统计或 API Key。

维护者发布前应运行 `npm run release:check`，确认测试、构建和 portable ZIP 校验通过。

## 开始前准备

你需要准备：

- Windows x64 系统。
- UTF-8、GBK 或常见中文编码的 TXT 小说。
- 一个可用的 AI 服务账号、API Key、Base URL 和模型 ID。
- 稳定网络，以及足够的模型额度。

支持的模型接口：

- OpenAI-compatible Chat Completions：适用于 OpenAI、DeepSeek、豆包 / 火山方舟、智谱、Moonshot / Kimi、MiniMax、小米 MiMo、SiliconFlow 等兼容服务。
- Anthropic Messages：适用于 Anthropic 官方 Claude，以及提供 Anthropic 兼容端点的 DeepSeek、豆包 / 火山方舟、智谱、Moonshot / Kimi、MiniMax、小米 MiMo 和 SiliconFlow。
- Gemini：使用 Google Gemini 原生接口。

不同服务商的模型 ID、价格、并发、TPM/RPM 限制和参数支持会变化，请以服务商当前文档为准。

## 快速上手

1. 点击“导入 TXT”，软件会自动识别章节，并按批次整理。
2. 先配置模型，填写 Base URL、模型 ID 和 API Key，保存后点击“诊断模型”。
3. 进入“设定”，填写主角姓名、改写后姓名、主角别名映射、其他女性化姓名映射、女主候选、外观模式和高级设定。
4. 如果章节识别不理想，可进入“章节规则”预览并调整规则；尚未开始分析或改写前可以重新生成章节列表，也可以为本次生成开启长章节自动拆分。
5. 在“设置”中按需要选择每批 10、30、50 或 100 章，以及 1、3、6、10、25 或 50 并发。高并发选项会根据批次大小自动限制。
6. 建议先处理一个批次：点击“分析”，再点击“改写”，确认效果稳定后再使用“一键分析改写”。
7. 改写复检默认开启，会增加请求数、等待时间和 Token 消耗；如需优先速度，可在设置中关闭。
8. 一键分析改写会按批次连续处理；运行中可暂停、继续或终止。继续时会保留当前批次已经完成的分片，只处理未完成分片。
9. 改写完成后进入“对比”页面，可搜索、查看差异、本地检查、手动编辑、单章重写并导出 TXT；导出只包含已完成改写的章节。

温馨提示：应用数据默认位于 `C:\Users\你的用户名\AppData\Roaming\com.local.yurirewrite`。删除该目录会清除本地小说、改写稿、配置和日志，只应在已经备份且确定不再需要这些数据时使用。

## 主要功能

### 章节识别

软件支持常见中文网文章节格式和连续数字标题。对于夹在连续正式章节之间的“第一更”“第二更到”“还有第三章”“未完待续”等作者更新提示，新导入时会保守识别为非章节内容；如果更新提示误占了正文，会在前后正式章节编号连续时把正文归回正确章节。番外、外传、后记、完本感言及无法确定的内容仍会保留。

导入后只要还没有开始分析或改写，可以回到“章节规则”页面重新预览并生成章节列表。开始过分析、改写、存在任务历史或手写一致性资产后，软件会禁止重切分，避免破坏章节 ID 与已有结果的对应关系。

章节规则页提供“长章节自动拆分”选项。开启后，本次预览或生成出的正式章节如果正文去空白后超过 5000 字，会自动拆成多个章节，标题追加“（1）”“（2）”等编号；未识别章节时按长度生成的分段不受这个选项影响。

### 批次处理

章节型小说可选择每批 10、30、50 或 100 章。修改设置后，软件会重新生成已导入章节型小说的批次范围，但不会删除章节、分析结果、改写稿、人工编辑或一致性资产。无法识别章节的小说继续按每段最多约 10 万字处理。

分析和改写共享并发设置。每批 10 或 30 章最高并发 10，每批 50 章最高并发 25，每批 100 章最高并发 50。程序会把有效章节均匀切成连续分片，空伪章节不参与分片。

工作区上方会显示当前任务的整体进度。单独“分析”“改写”和“一键分析改写”都会显示任务类型、阶段、章节完成数和分片进度；章节列表只显示章节状态，不在每章标题后追加进度条。

### 小说设定

“设定”是独立页面，分为基础设定、高级设定和设定预览。基础设定中可以配置主角原名、改写后姓名、主角别名、其他女性化姓名、女主候选 / 关系对象、身材体型和改写模式。

主角别名和其他女性化姓名都支持逐行映射：只填写原名时，由 AI 自动生成女性化名称；填写“原名 -> 改写后姓名”时，软件会把目标名作为强制姓名映射，后续分片、批次、复检修复和单章重写都会继续使用。主角别名的目标名不会被本地检查当作旧名残留。

女主候选用于提示模型加强相关百合互动和关系连续性，但不会覆盖未指定角色性别、章节边界或原文主线逻辑。高级设定适合填写全局文风、互动尺度、外貌描写偏好等长期要求；设定预览会在保存前汇总当前草稿。

### 原文分析

“分析”只整理原文设定，不会提前加入百合改写要求。分析结果包括人物、性别、称谓、别名、关系、地点、伏笔、术语和姓名映射等一致性资料。主角别名会作为同一人物参与识别和后续姓名映射。

### 改写

“改写”会结合小说设定、核心设定和一致性资料生成改写稿。主角姓名、代词、称谓、外貌、互动和关系推进会按设定处理；未指定性转的角色、动物、灵兽等默认保持原文性别和称谓。

改写提示词会按优先级组织规则：输出格式和章节 marker 最高，其后是核心设定、用户指定姓名映射、未指定角色性别保持、原文主线逻辑、创意补写、文风润色和保守清理。软件会要求模型修正明显错别字、乱码、广告和无关标点，但不得借此删除剧情正文、番外、后记、人物台词、专有名词或风格化表达。

### 模型参数

模型配置支持 Temperature、Top P 和按服务商/模型适配的思考模式。使用推荐模型时，界面会说明该模型是否支持开启或关闭思考；不确定或第三方反代服务建议使用“自动”。

“提示词模糊”默认关闭。开启后，发送给该模型的提示词会在请求前弱化已配置的敏感表达，用于降低部分服务商的安全拦截概率；它不会修改本地小说、改写稿、设定或日志原文，也不能保证完全避免服务商拦截。

### 改写复检

复检默认开启，会让审查模型检查改写稿，发现主角男性残留、缺句、重复、串章、逻辑冲突或设定不一致时，会要求改写模型修复。复检最多三次审查、两次修复，质量更稳，但更慢、更耗 token。

### 暂停与恢复

一键任务会记录当前批次已经完成的分析或改写分片。手动暂停、限流、网络故障、临时网关错误或模型格式错误导致暂停后，可以调整并发或模型再继续；程序会为剩余章节重新生成连续分片，并把已完成章节作为只读相邻上下文，减少重复调用和上下文断裂。

### 对比与导出

对比页可以查看左侧原文和右侧改写稿，支持全局搜索、原文/改写稿范围筛选、差异高亮、字数对比、章节完成状态和章节跳转。你可以手动编辑并保存改写稿，也可以：

- “重写本章（原文）”：使用原文、设定、一致性资产和相邻上下文重新生成。
- “重写本章（改写稿）”：以当前改写稿为主要底稿继续修改。
- “恢复初稿”：恢复第一次单章重写前保存的改写版本。

本地检查面板会用前端规则扫描当前改写稿，不调用模型、不写入数据库。它会提示旧名残留、主角附近男性称谓、可疑群体代词、广告/乱码、重复段落、异常空稿、疑似未改写和原文/改写稿字数差过大等问题。未开始改写的章节不会因为改写稿为空而刷屏；人工编辑稿和以改写稿为基础的单章重写稿不会触发字数差提示。

检查结果支持批量忽略当前问题，也可以逐条关闭某个问题。忽略记录只保存在本机浏览器存储中；后续新章节、新证据或新类型问题仍会正常显示。问题列表使用虚拟滚动，大量问题也能继续滚动查看。

导出只包含已经完成改写且内容非空的章节，不会用原文补齐失败章节。

### 日志与 Token 统计

日志页用于查看 AI 调用的提示词、思考过程、输出和服务商原始响应。Token 统计使用独立的本地记录；清空日志不会同时清空历史 Token 统计。

## 常见问题

### 模型诊断失败

优先检查 Base URL、模型 ID、API Key、账户额度和服务商地区限制。诊断结果会尽量保留服务商返回的错误信息。

### API 调用很慢

常见原因是网络不稳定、服务商排队、并发过高、模型思考时间过长或本地应用数据异常。可以先降低并发、关闭思考模式或换模型。如果仍然异常缓慢，可尝试删除本机应用数据目录 `C:\Users\你的用户名\AppData\Roaming\com.local.yurirewrite` 后重新配置。

### 出现 HTTP 429、TPM 或 RPM 限流

这是服务商限制请求数或 Token 速率。程序会自动等待并降低有效并发；重试耗尽后，一键任务会暂停。你可以降低并发、等待额度恢复、关闭复检或切换模型后点击继续，已完成分片不会重做。

### 出现 HTTP 502、503、504 或 524

这通常表示模型服务或反向代理暂时不可用，并不一定是提示词或模型参数错误。普通任务会保留服务商响应并失败；一键任务会保存检查点后暂停。恢复服务或更换模型、代理后可以继续未完成分片。

### 审查决策 JSON 无法解析

这通常是复检模型没有稳定输出合法 JSON。程序会提示可以手动重试。频繁出现时，建议为复检选择 JSON 输出更稳定的模型，或降低并发后再试。

### 出现章节 marker 或输出截断错误

程序依靠稳定的章节开始/结束标记把批量回复拆回章节。模型输出不完整时会自动重试并缩小分片范围；如果仍失败，程序不会把无法可靠解析的内容写入章节。可尝试降低并发、减少处理范围或换用最大输出更高的模型。

### 导入后仍有更新提示伪章节

新的上下文识别规则只在导入时生效，不会自动重切已经导入并可能存在分析、改写或人工编辑的数据。请保留旧数据备份后重新导入原始 TXT，以使用最新章节识别规则。

### 累计 TXT 被占用

如果一键任务更新累计 TXT 时发现文件被阅读器或编辑器占用，程序会提示你手动关闭占用文件，然后继续重试。程序不会自动关闭外部软件。

## 本地数据与隐私

- 小说、批次、设定、改写稿、日志、Token 统计和导出文件保存在本机。
- API Key 优先保存到 Windows Credential Manager；系统凭据库不可用时才使用 SQLite 明文兜底，并会在界面提示。
- AI 日志可能包含提示词、小说片段、模型输出和服务商响应，但不会记录 API Key 或 Authorization 请求头。
- 调用在线模型时，相关小说片段会发送给你选择的 AI 服务商，请先确认其隐私政策和使用条款。
- 删除小说前，程序会列出将一起删除的数据并要求确认。

重要小说请自行保留原始 TXT 和导出的改写 TXT。
